# curl_cffi Feasibility Assessment

> **Staleness note:** References to "Gmail MFA" and "email login" below are historical. As of step 16f (April 2026), login uses MileagePlus number only, and MFA defaults to SMS.

## Verdict: NOT VIABLE for FetchAwardCalendar

curl_cffi with Chrome TLS impersonation bypasses Cloudflare for public pages and some API endpoints (ShopValidate), but **cannot reliably access the authenticated FetchAwardCalendar endpoint**. United uses Akamai Bot Manager (not just Cloudflare) on internal API endpoints, which requires browser-executed JavaScript sensor data that curl_cffi cannot provide.

**Architecture Decision: Playwright required** (see Architecture section below).

## Experiment Results

| Experiment | Result | Notes |
|---|---|---|
| 1 -- Single API call (no cookies) | FAIL | HTTP/2 INTERNAL_ERROR stream reset. Akamai rejects requests without valid `_abck` cookie |
| 1 -- Single API call (with Chrome cookies) | PASS | 200 OK, 30 days data, 126 records across all cabins |
| 2 -- Minimum viable request | PASS | Bearer token + cookies + minimal headers sufficient |
| 3 -- Rate & reliability (10 calls, 7s delay) | FAIL | 3/10 success. Akamai burns cookies after ~3-4 FetchAwardCalendar calls |

### Experiment 1 -- Single API Call

**Without cookies:**
- **Status:** FAIL
- **Behavior:** HTTP/2 INTERNAL_ERROR stream reset
- **Root cause:** Akamai Bot Manager rejects requests that lack a valid `_abck` cookie, regardless of TLS fingerprint
- **Impersonation target used:** chrome131
- **Cloudflare TLS check:** Passed (Cloudflare is not the blocker)

**With cookies exported from Chrome DevTools:**
- **Status:** PASS
- **API response:** Valid JSON with pricing data
- **Days with pricing data:** 30/30
- **Records returned:** 126 across all cabins (Economy, Business, First)

### Experiment 2 -- Minimum Viable Request

- **Status:** PASS
- **Minimum required configuration:** Bearer token (`x-authorization-api`) + Akamai cookies (`_abck`, `bm_sz`, `bm_sv`) + minimal headers
- **Cookies required:** Yes -- Akamai cookies are mandatory, not optional
- **sec-* headers:** Not strictly required when valid cookies are present
- **Origin/Referer:** Not strictly required when valid cookies are present

### Experiment 3 -- Rate & Reliability

- **Status:** FAIL
- **Success rate:** 3/10
- **Request interval:** 7 seconds between calls
- **Failure mode:** Akamai burns (invalidates) the `_abck` cookie after approximately 3-4 FetchAwardCalendar calls
- **Pattern:** First 3 calls succeed, then subsequent calls fail with stream resets or empty responses
- **No 403 (Cloudflare) responses observed** -- Cloudflare is not the problem
- **No 429 (Rate limit) responses observed** -- this is not traditional rate limiting
- **Conclusion:** Static cookies from Chrome DevTools work initially but are consumed/invalidated by Akamai after a small number of uses

## Key Discoveries

### Dual-layer bot protection

United uses **two independent bot protection systems**:

1. **Cloudflare** -- TLS fingerprinting at the edge. curl_cffi's Chrome impersonation handles this easily.
2. **Akamai Bot Manager** -- JavaScript sensor data validation via `_abck`, `bm_sz`, and `bm_sv` cookies. These cookies are generated and maintained by browser-executed JavaScript. curl_cffi **cannot** generate or refresh these cookies.

### Endpoint protection levels vary significantly

| Endpoint | Protection Level | curl_cffi Viable? |
|---|---|---|
| ShopValidate | None (no bot protection) | Yes -- works freely |
| FetchFlights | Medium (429 challenge response) | Unreliable |
| FetchAwardCalendar | Aggressive (HTTP/2 stream reset without valid cookies; burns cookies after ~3-4 calls) | No |

### Akamai cookie lifecycle

- The `_abck` cookie is generated by Akamai's JavaScript sensor running in the browser
- The sensor continuously updates the cookie value based on browser behavior (mouse movements, page interactions, JavaScript execution environment)
- A static cookie exported from Chrome DevTools works for a handful of API calls, then Akamai invalidates it server-side
- There is no way to refresh `_abck` without executing Akamai's JavaScript in a real browser environment

### Scraperly difficulty rating is misleading

Scraperly rates United at 1-2/5 difficulty. This applies only to **public-facing pages** (flight search results, etc.). Authenticated API endpoints like FetchAwardCalendar have significantly stronger protection that is not reflected in that rating.

## Architecture Decision: Hybrid — curl_cffi + Playwright Cookie Farm

curl_cffi cannot maintain Akamai sensor data on its own, but it doesn't need to. The proven approach is a **hybrid architecture**: Playwright runs in the background as a "cookie farm" maintaining fresh Akamai cookies, while curl_cffi makes the actual API calls.

```
┌─────────────────────────────────────────┐
│           Cookie Farm (Playwright)       │
│  Real Chrome browser running in bg       │
│  ├─ Akamai JS sensor executing          │
│  ├─ _abck cookie refreshing             │
│  └─ Exports cookies on demand            │
└──────────────┬──────────────────────────┘
               │ fresh cookies
               ▼
┌─────────────────────────────────────────┐
│         API Caller (curl_cffi)           │
│  ├─ Pulls cookies from Playwright        │
│  ├─ Makes FetchAwardCalendar POST        │
│  ├─ Detects cookie burn (stream reset)   │
│  └─ Requests cookie refresh on burn      │
└─────────────────────────────────────────┘
```

**Pure Playwright remains a fallback** if the hybrid approach has issues, as it has near-100% reliability (tested in `test_playwright.py`).

**Playwright persistent browser context** is used for the cookie farm:

- Playwright runs a real Chromium browser, so Akamai's JavaScript sensor executes naturally and keeps cookies fresh automatically
- Persistent browser context preserves session state (cookies, localStorage) across script runs
- Manual login with Gmail MFA is required on the first run; the session persists for hours afterward
- Subsequent runs reuse the persisted session without requiring login

**What this changes from the original plan:**

| Aspect | Original Plan (curl_cffi) | Actual (Playwright) |
|---|---|---|
| Dependencies | curl_cffi, python-dotenv | playwright, python-dotenv |
| Browser required | No | Yes (headless Chromium) |
| Resource usage | Negligible | Moderate (~200-400 MB RAM for headless browser) |
| Auth token handling | Manual copy from Chrome DevTools to `.env` | Playwright persistent context; manual login on first run only |
| Akamai cookies | Manual export (burns after ~3-4 calls) | Automatically maintained by browser |
| Reliability | ~30% success rate | Expected near-100% (browser is indistinguishable from real user) |
| Infrastructure | Laptop or minimal VPS | Laptop or VPS with enough RAM for headless browser |
| **Hybrid (cookie farm)** | curl_cffi + playwright | Yes (background, for cookies only) | ~450 MB (browser + Python) | Playwright persistent context; manual login on first run | Automatically maintained by cookie farm, refreshed every 2 calls | Target 90%+ (curl_cffi speed + Playwright cookie freshness) | Laptop or VPS |

## Authentication Decision (Updated)

Manual login remains the approach for Phase 1, but the mechanism changes:

- **Before (curl_cffi plan):** User logs into Chrome, copies bearer token to `.env`, copies cookies manually
- **After (Playwright plan):** User runs the scraper, Playwright opens a browser window for manual login with Gmail MFA on first run. Session persists in Playwright's user data directory. Subsequent runs skip login entirely until the session expires.

This is actually **less manual effort** than the original curl_cffi plan, since the user no longer needs to export cookies or tokens from Chrome DevTools.

## Open Questions / Future Work

- **Session duration:** How long does a Playwright persistent session last before United requires re-authentication? To be measured during burn-in testing.
- **Headless vs headed:** Whether Playwright headless mode triggers additional bot detection. May need to run headed on first login, then switch to headless for automated sweeps.
- **Automated auth:** Playwright + IMAP for Gmail MFA codes -- deferred to Phase 1b.
- **Rate limits under Playwright:** Whether United tolerates faster intervals when requests come from a real browser session. The 7-second interval may be overly conservative.
- **Resource optimization:** Whether a single Playwright browser context can be shared across parallel route queries, or if sequential execution is required.
